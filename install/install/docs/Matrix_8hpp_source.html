3  Design Decision -- Eigen Integration:
4 
5  At the very beginning, we used vector/matrices from libminisl. Then we
6  began to switch to Eigen's tools (because already used in lots of our code).
7  Thus during the migration phase, we used custom class inheriting from
8  Eigen::Matrix.
9  However I encountered issues when executing the code under linux. It
10  appears it was because of SSE instructions (special pipeline on CPU allowing
11  to perform some vector operations in parallel). This SSE instruction required
12  128-bit aligned memory. There are articles about this on Eigen website:
13  eigen.tuxfamily.org/dox/group__DenseMatrixManipulation__Alignement.html
14 
15  But to summerize: keeping the alignment is difficult because you have to
16  overload new operator in each class containing an Eigen::Matrix. Too unsafe,
17  thus I disable this (Eigen::DontAlign) but missing assignment operators did
18  that this consideration was ignored in some cases.
19  E.g.: sibr::Matrix A, B, C;
20  // ... // set A and B
21  C = A*B; // A*B return a temporary class of Eigen but C didn't have the
22  // assignment operator for this class [...] it wrongly considered it has an
23  // not-aligned matrix and data was corrupted.
24 
25  Now SIBR uses a plugin system to extend Eigen classes:
26  eigen.tuxfamily.org/dox/TopicCustomizingEigen.html
27 
28  It's both safer and faster. (but it was not possible during the migration
29  phase because I needed the child type to perfom automatic convertion with
30  remaining libminisl tools).
